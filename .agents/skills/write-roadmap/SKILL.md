# 编写路线图

在保证准确、可学习和可验证的前提下，用最少的上下文完成 `src/content` 中的 YAML 路线图工作。

## 省 token 工作流

1. 先用 `rg --files src/content` 和 `rg` 定位目标、同类路线图、已有 ID 与引用；只读取目标文件和最接近的 1–2 个范例。
2. 仅在新增结构、字段含义不明或校验失败时读取 `src/types.ts`、`src/data.ts` 的相关片段；仅在项目命令或目录约定不明时读取 `README.md`，不要默认通读。
3. 从用户请求和现有内容推断学习者、起点、终点与边界；只有不同选择会显著改变路线图时才提问。
4. 先搜索是否可用 `type: roadmap` 复用已有路线图。目标和深度相近时复用，不复制节点。
5. 只研究本次新增或改变的知识点和链接。稳定且确定的内容无需广泛检索；易变化、版本相关或不确定的内容批量联网核实，优先官方文档、标准和维护方资料。
6. 新建或大幅重构时，读取 [格式与质量标准](references/roadmap-guidelines.md)，先列简短骨架再写 YAML；小范围文案、状态或链接修改无需加载该文件。
7. 做最小改动并检查 diff。完成全部编辑后只运行一次 `npm test` 和 `npm run build`；若失败，只读取相关错误和代码片段并修复本次引入的问题。

## 核心约束

- 用可验证的能力或成果命名节点，并用一句话描述学习范围或完成标准。
- 主线按依赖关系组织，通常覆盖基础、核心技能、工程实践、综合项目和深化方向；只在真实从属关系中使用 `children`。
- 新核心节点使用 `planned`，非必需分支使用 `optional`；不要猜测用户进度。
- 把测试、安全、调试、性能、可访问性或可观测性放入相关阶段，并至少包含一个综合实践。
- 链接直接指向匹配主题的权威入口；不编造链接，不凭记忆声称“最新”。
- 延续中文文风，保留技术专名官方拼写；ID 使用稳定、简短、语义化的 kebab-case 英文。

## 修改既有路线图

- 保留无关节点、状态和已有 ID；仅在语义改变时重命名，不为整齐而重排。
- 先判断缺失内容应是节点、子节点还是路线图引用。
- 更新链接时只核对被修改的 URL，且只允许 `http` 或 `https`。

## 交付

简要说明改动、关键范围假设、复用引用和验证结果；不要复述完整路线图。若自行选择了会影响范围的技术栈，明确标注。
