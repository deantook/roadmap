# 编写路线图

为学习者设计可执行的能力成长路径，并以本仓库接受的 YAML 写入 `src/content`。兼顾知识依赖、实践闭环和内容准确性；不要把路线图写成关键词清单。

## 工作流

1. 读取 `README.md`、`src/types.ts`、`src/data.ts` 和相关的现有 YAML。以代码中的 schema 为最终格式依据。
2. 明确目标学习者、起点、终点和主题边界。用户未说明时，假定读者具有基础计算机操作能力，目标是从入门成长到能独立完成小型生产级项目；在结果中简短说明这一假设。
3. 检查现有路线图是否已覆盖该主题。能用 `type: roadmap` 复用时不要复制一套节点；只有目标与深度明显不同才新增路线图。
4. 研究主题。优先使用官方文档、标准、规范和项目维护方资料；对于会变化的技术、工具版本或推荐实践，先联网核实。不要编造链接或凭记忆指定“最新”版本。
5. 先设计学习骨架，再写 YAML。按“基础概念 → 核心技能 → 工程实践 → 真实项目 → 深化方向”组织；只在确有从属或前置关系时使用 `children`。
6. 写入或修改 `src/content/<roadmap-id>.yaml`。遵守 [格式与质量标准](references/roadmap-guidelines.md)。
7. 自检学习路径：节点是否可学习、顺序是否合理、是否包含动手实践与质量保障、可选内容是否与主线分离、引用是否真的减少重复。
8. 运行 `npm test` 和 `npm run build`。修复由本次改动造成的 schema、重复 ID、缺失引用、URL 或构建错误。

## 内容设计规则

- 用能力或学习成果命名节点，例如“使用事务维护一致性”，避免只写“事务”。
- 每个描述用一句话说明学习范围或完成标准，不重复标题，不写空泛的“了解相关知识”。
- 保持主线克制。优先设计 6–12 个顶层阶段；每个阶段通常包含 2–6 个子节点。根据主题复杂度调整，不为凑数拆分。
- 将安全、测试、调试、性能、可访问性或可观测性放进相关阶段，不要一律堆到末尾。
- 至少提供一个综合实践节点，使学习者能把前置知识组合成可验证的产物。
- 对多条同样合理的分支使用 `optional`，不要伪造唯一顺序。
- 新内容的核心节点使用 `planned`，选修节点使用 `optional`。不要推测用户已经完成或正在学习哪些内容。
- 链接应直接指向与节点匹配的权威学习入口。宁缺毋滥；不要链接搜索结果页、营销软文或不稳定的聚合页。
- 延续仓库现有中文文风；技术专名保留官方拼写。ID 使用稳定、简短、语义化的 kebab-case 英文。

## 修改既有路线图

- 保留无关节点、用户进度默认值和已有 ID；仅在语义确实改变时重命名 ID。
- 先判断缺失内容应成为节点、子节点还是独立路线图引用。
- 不因追求对称而重排整棵树。若顺序变化会改变学习含义，在交付说明中指出。
- 更新外部链接时核对目标页面的主题、维护方和协议；只允许 `http` 或 `https`。

## 交付

简要说明新增或调整的路线图、关键范围假设、复用的路线图引用，以及验证结果。若因缺少用户选择而采用了特定技术栈，明确标注该选择。
